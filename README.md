## Описание проекта
Одна из самых частых операций при написании JS-кода на фронтенде — это поиск элементов в DOM. При этом важно не только правильно найти элементы, но и грамотно переиспользовать те, которые уже были найдены ранее. Это позволит не тратить впустую процессорное время и память устройства, с которого пользователь обратился к нашему веб-приложению.
В этой проектной работе частично воспроизведена работа одного из видов приложений — парсеров.
### Дано:
В стартовом пакете дана сверстанная страница и подключённый к ней сценарий. Он выводит содержимое страницы в консоль в виде объекта при помощи вызова функции parsePage(). Она возвращает уже подготовленный объект ответа, но пока без полезной информации. В задании также был представлен образец информации в объеме и виде, которые ожидаются при выводе результатов работы.
### Задача:
Заполнить представленную структуру информацией со страницы.

## Инструкция по запуску
Для запуска откройте `index.html`.
Работа выполнялась в файле parser.js.
Для проверки работы вызов makeTests() должен быть раскомментирован.

## Что было реализовано в ходе работы
В процессе выполнения проекта были:
- Использованы знания и навыки, полученные при изучении свойств JS, который может работать не напрямую с HTML, а с DOM (Document Object Model) объектной моделью документа, которую создает браузер
- Для изменения некоторых элементов с целью подготовки для конечного вывода во избежание изменения оригинальной страницы применен 'cloneNode' для клонирования элемента в память
- Использованы стандартные методы поиска элементов: по селектору, по массиву (псевдомассив NodeList), свойство children c псевдомассивом, который можно использовать в циклах и обращаться к элементам по индексам.
- Использованы методы атрибутов 'getAttribute', 'removeAttribute'
- Работа со свойствами - получение имени класса через свойство `className' и управление списком классов через свойство 'classList', для проверки наличия класса у элемента - использовался метод 'contains'
- Работа с 'dataset' с извлечением и использованием 'data-атрибутов'
- Для получения текстового содержимого элемента использовано свойство 'textContent'. Для получения всего наполнениния элемента, включая разметку, использовано свойство innerHTML
- При работе с текстовым содержимом не было забыто о возможны лишних пробельных символах, для избавления от которых использовался метод 'trim()'
- Использовались ранее изученые конструкты и методы JS такие как 'split()', 'forEach', 'switch-case', 'toFixed', циклы 'for', навыки работы с объектами и списками

## Рефакторинг
При дальнейшей работе были сделаны следующие улучшения:
- Объекты 'metaObject', 'openGraphObject', 'productObject' изначально были созданы без ключевых слов 'const' или 'let', что является плохой практикой, т.к. делает их свойствами глобального объекта 'window' и может в перспективе приводить к багам, если на странице несколько скриптов. Также принято решение переместить весь код в функцию parsePage с объявлением основных переменных в начале. Цель - "изоляция" - переменные живут только внутри функции и никому не мешают, "чистота" - при вызове функции два раза подряд каждый раз будут создаваться чистые пустые объекты.
- два дубля конструкции switch(символ->букв.код валюты) заменены и объеденены в функцию, куда передается проверяемое значение и объект со всеми вариантами соответствий, а также дефолтное значение.
- заменил "отбрасывание" значка валюты '.split(currencySymbol)[1]' на регулярное веражение '.replace(/\D/g, '')' - на случай, если в строке окажутся два одинаковых символа, то метод 'split' даст некорректный результат.
- изменил "хрупкий" код, где через 'split' из 'textContent' div-а с собственным вложенным текстом и текстом вложенного 'span', на вариант с 'firstChild' - поскольку для этого свойства характерно то, что оно расценивает как "ребенка" в том числе и вложенный текстовый узел.
- прогнал код через форматтеры.
- добавил в проект файл 'example.js' - с образцом того, что должно получаться на выходе по ТЗ
